# `compute_kappa` — implementation notes

Companion to `correlated_noise_correction.md`, which explains *why* we need
a correction factor `kappa` on the diagonal-Fisher flux error when pixels
are correlated. This doc explains *how* `compute_kappa` is implemented in
`slimfarmer/image.py:550`, and — just as importantly — how it avoids
blowing up memory when you have thousands of sources and five bands.

Source of truth: `slimfarmer/image.py` (`FarmerImage.compute_kappa`) and
`slimfarmer/_group.py` (`SourceGroup._cache_kappa_data`).

---

## 1. What we're computing, in one line

For each source `s` and band `b`:

```
kappa_b(s) = sqrt( max( h^T R h / D , 1 ) )
```

where (all per source, per band):

- `T(y,x)` is the unit template — the PSF-convolved source profile,
  **normalised** so that if you scaled it by `flux` you'd recover the
  model image. In the code this is `unit_t = engine.getModelImage() / flux`.
- `h(y,x) = sqrt(invvar_bg(y,x)) * T(y,x)` — the template weighted by the
  independent-pixel noise, on the *background-only* inverse-variance.
- `D = sum_i invvar_bg_i * T_i^2` — the diagonal-Fisher denominator (the
  quantity that would give you the flux error if the pixels were
  independent: `flux_err_diag = 1/sqrt(D)`).
- `R(dy, dx)` is the 2D pixel-noise correlation function, estimated from
  the noise realizations and normalised so that `R(0, 0) = 1` (it lives
  in `band_config[band]['noise_corr']`).

The result is stored on each source model as
`model.flux_err_noisereal_kappa[band]`, and used downstream to rescale
`flux_err_des`.

`kappa == 1` means pixels are uncorrelated → no correction. For Dec25
IMCOM coadds kappa ends up ~5.

---

## 2. Why the naive implementation is expensive

`h^T R h` is the contraction of a vector with a matrix. If you wrote it
literally:

```python
kappa_sq = h.flatten() @ R_full @ h.flatten()    # DON'T DO THIS
```

then `R_full` is an `(Npix × Npix)` matrix. For an IMCOM block of
`2108²` ≈ 4.4 M pixels, that's 4.4 M × 4.4 M ≈ 1.9 × 10¹³ entries —
about **150 TB in float64**. Completely unusable. Two structural
observations let us avoid ever materialising it:

**(a) R is translation-invariant.** Background noise statistics don't
depend on where you are in the image, so `Cov(pixel_j, pixel_k)`
depends only on the offset `(dy, dx) = k − j`, not on `j` and `k`
separately. All the information in that huge matrix fits in a 2D array
of the same shape as one image: the correlation function `r(dy, dx)`.
That's `noise_corr`, stored once per band under
`band_config[band]['noise_corr']`.

**(b) Translation-invariant matrix-vector products are convolutions.**
Given translation-invariance, the quadratic form simplifies to

```
h^T R h = sum_{y,x} h(y,x) * (R * h)(y,x)
```

where `R * h` denotes 2D convolution. Convolutions are cheap via FFT:

```python
Rk = rfft2(noise_corr)
Hk = rfft2(H)
conv = irfft2(Rk * Hk, s=(ny, nx))
hTRh = (H * conv).sum()
```

That replaces the `O(Npix²)` matmul with `O(Npix * log Npix)` FFTs on
arrays of *one-image* size. This is the core of why `compute_kappa` is
tractable at all.

## 3. How `R` (`noise_corr`) is estimated from the noise realisations

`R(dy, dx)` is not a theoretical quantity — we measure it empirically
from the per-band **noise realisation cube** that IMCOM writes out
alongside the coadded science image. The cube
`roman_noise_reals.fits` is shape `(n_real, ny, nx)`: `n_real`
independent draws of the background noise field on the same pixel
grid as the science image, generated by propagating IMCOM's linear
coadd through independent instances of the underlying instrument
noise. Each slice is exactly what the background of the science image
*would* have been if that particular noise draw had been the one on
sky.

The full estimator lives in `FarmerImage.__init__` at `image.py:126`:

```python
noise_corr = None
if noise_reals_path is not None:
    ny_f, nx_f = sci.shape

    # (1) per-pixel sigma from the ensemble, smoothed
    with fits.open(noise_reals_path) as hdul:
        nr_all = hdul[0].data.astype(np.float64)
    sigma_nr = np.std(nr_all, axis=0, ddof=1)
    from scipy.ndimage import uniform_filter
    sigma_nr = uniform_filter(sigma_nr, size=15)
    sigma_nr = np.where(sigma_nr > 0, sigma_nr, 1.0)

    # (2) averaged power spectrum of the normalised realisations
    n_real = nr_all.shape[0]
    R_2d = np.zeros((ny_f, nx_f), dtype=np.float64)
    for k in range(n_real):
        n_norm = nr_all[k] / sigma_nr
        R_2d += np.abs(np.fft.fft2(n_norm)) ** 2
        del n_norm
    R_2d /= n_real

    # (3) inverse FFT → real-space correlation, with the FFT normalisation
    noise_corr = np.fft.ifft2(R_2d).real / (ny_f * nx_f)
    del R_2d, nr_all, sigma_nr
```

Three conceptual steps.

### (1) Per-pixel sigma from the realisations themselves, then smooth

`sigma_nr[y, x] = std_k(nr_all[k, y, x], ddof=1)` — for each pixel,
the standard deviation *across* the `n_real` noise draws. This is the
correct per-pixel background noise level by construction, and it
avoids a unit-mismatch trap: slimfarmer also carries an `invvar`
(weight) map, but that map uses whatever convention the coadder chose
(sometimes ADU, sometimes e⁻, sometimes a scaled variance), and
calibrating `invvar → sigma` across different reductions is a
maintenance hazard. Going straight to the realisations sidesteps all
of that — whatever unit the noise is in, `std` gives the right sigma
in those same units.

A uniform-box smooth (`uniform_filter(size=15)`) is applied because
with only a handful of realisations (Dec25 has ~50 per band) the
per-pixel std is itself noisy, and physically the depth map varies
slowly — `sigma` really should be smooth at the scale of a few
arcseconds. The assumption we're making is that sigma varies on
scales ≫15 pixels; sharp edges in the sigma field (mask boundaries,
chip gaps) are slightly blurred by the smoothing, which is acceptable
in the interior of the footprint. Dead pixels where `sigma_nr == 0`
are replaced by `1.0` so the division in step (2) doesn't NaN — those
pixels end up with `h ≈ 0` downstream anyway because their
`invvar_bg` is also zero.

### (2) Average power spectrum of the normalised noise field

After dividing each realisation by sigma, the normalised field
`n_norm = n / sigma` is **unit-variance** everywhere by construction.
If you believe stationarity (sigma captures all the spatial variation
in amplitude, and only the *correlation structure* stays), then
`n_norm` is a stationary Gaussian field with `<n_norm(x)> = 0`,
`<n_norm(x)²> = 1`, and

```
<n_norm(x) n_norm(x + d)> = r(d)      (by definition of R)
```

The **Wiener–Khinchin theorem** says that for a stationary process,
the autocorrelation and the power spectrum are a Fourier-transform
pair:

```
r(d)  =  F^{-1}( |F(n_norm)|² )
```

(modulo FFT normalisation conventions; see step 3). So we estimate
`|F(n_norm)|²` per realisation and average it across the `n_real`
draws to get a Monte-Carlo estimate of the power spectrum.

```python
R_2d += np.abs(np.fft.fft2(n_norm)) ** 2
...
R_2d /= n_real
```

Averaging `n_real` realisations drops the per-frequency estimator
variance by `1/n_real`; with ~50 realisations the power spectrum is
known to ~14% per Fourier mode, which is fine because we only care
about the correlation at small lags where nearby modes get averaged
together again by the inverse FFT. Looping over realisations rather
than building a `(n_real, ny, nx)` intermediate keeps the peak memory
to one image — the loop also `del n_norm`s each iteration so
temporaries don't pile up.

### (3) Inverse FFT with the FFT normalisation factor

```python
noise_corr = np.fft.ifft2(R_2d).real / (ny_f * nx_f)
```

Two things to understand.

**Why `.real`.** The Fourier transform of an autocorrelation (a real,
even function) is real-valued, so mathematically `ifft2(R_2d)` should
have no imaginary part. Numerically there's floating-point noise at
the 10⁻¹⁵ level; `.real` drops it.

**Why divide by `ny_f * nx_f`.** This is the numpy FFT convention.
`np.fft.fft2` has no scaling, `np.fft.ifft2` divides by `N = ny*nx`.
Parseval's identity in numpy form is

```
sum |n_norm|²  =  (1/N) · sum |fft2(n_norm)|²
```

Plugging in `<|n_norm|²> ≈ 1` (unit variance) gives
`sum |fft2(n_norm)|² ≈ N²`, i.e. the power spectrum has overall scale
`~N` after the `ifft2` normalisation alone. To get `r(0, 0) = 1`
(the definition of an autocorrelation: zero lag = variance, which is
1 because we normalised), we need an extra division by `N`. So:

```
r = ifft2(|fft2(n_norm)|² averaged) / N
```

with `N = ny_f * nx_f`. Equivalently: `np.fft.fft2` + `np.fft.ifft2`
round-trip through a factor of `N`, so computing an autocorrelation
via the power spectrum carries exactly one extra `/N`.

After this line you can verify `noise_corr[0, 0] ≈ 1` (the central
pixel is the variance of `n_norm`, which is unity by construction),
and `noise_corr` falls off to ~0 within ~10 pixels — that falloff
envelope *is* R, and it's the whole reason kappa is larger than 1.

### What `noise_corr` looks like in practice

For Dec25 IMCOM coadds at the default 0.049" pixel scale,
`noise_corr` is a broad positive blob of FWHM ~5–8 pixels, with
`noise_corr[0, 0] = 1` by construction and typical off-diagonal values
of 0.3–0.6 at lag 1 pixel, dropping to the noise floor beyond ~15
pixels. It's symmetric under `(dy, dx) → (−dy, −dx)` (real, even
autocorrelation) and nearly rotationally symmetric (IMCOM's output
kernel is close to isotropic). This is the object you `rfft2()` once
per band at the top of `compute_kappa`'s outer loop.

### What this relies on (and where it can break)

- **Stationarity of the correlation.** We factor the noise field as
  `n(x) = sigma(x) · n_norm(x)` with `n_norm` stationary. If the
  *correlation length* itself varies across the image (e.g. near
  chip edges or mask boundaries where IMCOM's kernel changes shape),
  a single `noise_corr` is an approximation. In practice it's a good
  one for the IMCOM interior; the outer 34-pixel overlap skirt is
  where you'd expect it to be worst.
- **Enough realisations.** With `n_real = 1` you'd have no averaging
  and the estimator would equal the per-realisation periodogram,
  which is pure noise at each frequency. Dec25 has ~50 realisations
  per band — enough for kappa to be well-defined and for per-tile
  variations to be small.
- **Sigma smoothing matches reality.** The 15-pixel uniform filter
  assumes the depth map is smooth at the arcsecond scale. If you
  feed in data with genuinely sharp sigma features (hot pixels,
  bright bleeds) the smoothing will wash them out and `noise_corr`
  will effectively absorb the residual non-stationarity as spurious
  "correlation". This is why the masks zero those pixels out of
  `invvar_bg` upstream — the kappa loop then ignores them regardless.
- **The noise is close to Gaussian.** Wiener–Khinchin is true for any
  second-order stationary process, so we don't strictly need
  Gaussianity for the `r(d) = F^{-1}(P(k))` identity — but for
  kappa to be interpretable as a simple flux-error scale factor, the
  underlying noise should be close to Gaussian. IMCOM's output is a
  linear combination of many inputs, so the central-limit-theorem
  argument is strong.

## 4. The per-source cache (the main memory trick)

Even after (a) and (b), storing the full 2D template `H` for every
source is still wasteful. `H` has the same shape as the image (call it
`ny × nx`), but is **zero almost everywhere** — a galaxy's PSF-convolved
profile touches maybe ~few-hundred pixels out of millions. So instead of
pickling a dense `(ny, nx)` template per source (that would be tens of
megabytes per source times thousands of sources — easily hundreds of GB
per tile), the group-fitting code caches only the nonzero values and
their indices.

This happens in `SourceGroup._cache_kappa_data` (`_group.py:837`):

```python
src_copy = copy.deepcopy(model)
temp_engine = Tractor([img], [src_copy])
temp_engine.bands = [band]
temp_engine.freezeParam('images')
unit_t = temp_engine.getModelImage(0) / flux     # normalised template

D = float(np.sum(invvar_bg * unit_t ** 2))        # scalar
h = np.sqrt(invvar_bg) * unit_t                   # 2D but mostly zero
nzy, nzx = np.nonzero(h)                          # short index arrays

model._kappa_cache[band] = {
    'h_vals': h[nzy, nzx].astype(np.float32),     # only the nonzero values
    'nzy':    nzy.astype(np.int16),               # y-coordinate of each
    'nzx':    nzx.astype(np.int16),               # x-coordinate of each
    'D':      D,                                  # scalar
}
```

What gets saved per source per band:

| field    | shape       | dtype    | typical size           |
|----------|-------------|----------|------------------------|
| `h_vals` | `(Nnz,)`    | float32  | a few KB (Nnz ~ 10³)   |
| `nzy`    | `(Nnz,)`    | int16    | 2× smaller than int64  |
| `nzx`    | `(Nnz,)`    | int16    |                        |
| `D`      | scalar      | float64  | 8 bytes                |

`int16` here is safe because IMCOM blocks are 2108 px on a side, which
fits easily in ±32767. Using `int16` instead of `int64` cuts the index
memory by 4×.

Contrast with the alternative of storing the full `(ny, nx)` `h` array:
float64 at 2354 × 2354 = **44 MB per source per band**. Over ~1000
sources × 5 bands that's ~220 GB, which you cannot pickle to a worker,
let alone keep in RAM. The sparse cache typically comes out to a few KB
per source per band — a ~10⁴× reduction.

### Which pixels end up in the cache?

Not "all pixels the PSF touches" — the footprint is specifically
**the dilated groupmap**, set up in `_stage_images` (`_group.py:149`):

```python
if self.group_id in self.groupmap_local:
    gy, gx = self.groupmap_local[self.group_id]
    fit_mask = np.zeros(masked.shape, dtype=bool)
    fit_mask[gy, gx] = True                            # start from groupmap
    ...
    fit_mask = binary_dilation(fit_mask, structure=struct)  # dilate by fit_dilation_radius
    masked[~fit_mask] = 1.

weight[~np.isfinite(data) | (masked == 1) | ~np.isfinite(masked)] = 0.
...
self.band_data[band]['invvar_bg'] = weight.copy()
```

`fit_dilation_radius` is `0.2"` in the default `Config` — ~4 Roman
pixels, a small skirt around the seg-map to catch profile wings. Every
pixel outside that mask gets `weight = 0`, so `invvar_bg = 0` there too.
Then in the cache step:

```python
h = np.sqrt(invvar_bg) * unit_t
nzy, nzx = np.nonzero(h)
```

Because `invvar_bg` is zero outside the dilated groupmap, `h` is zero
there regardless of what `unit_t` looks like — even though the
PSF-convolved template carries flux far beyond the seg-map. So
`nzy, nzx` only live inside the dilated groupmap footprint, typically
a few hundred pixels for a compact source up to a few thousand for a
big group.

This is the **correct** footprint to use: the diagonal-Fisher
denominator

```python
D = float(np.sum(invvar_bg * unit_t ** 2))
```

uses the *same* masked `invvar_bg`, so `kappa² = h^T R h / D` is a ratio
between two quantities integrated over identical pixel sets. If we
evaluated the numerator over a wider footprint than the denominator,
the ratio wouldn't mean anything. The wings outside the seg-map
contribute to neither, which is consistent.

The comment on the method says it explicitly:

> Only stores 1D arrays (h_vals, nzy, nzx) and scalar D — NOT the large
> pairwise dy/dx matrices, to keep pickle size small.

"Pickle size" matters because fitted group results are shipped back from
worker processes to the main process, so anything attached to the model
objects has to round-trip through pickle. Keeping `_kappa_cache` tiny
keeps that cheap.

## 5. The `compute_kappa` loop itself

Once every source has its tiny `_kappa_cache`, the main-process loop in
`FarmerImage.compute_kappa` reconstructs `H` on the fly and evaluates
the convolution:

```python
def compute_kappa(self):
    import gc

    for band in self.bands:
        bc = self.band_config[band]
        noise_corr = bc.get('noise_corr')
        if noise_corr is None:
            continue
        ny, nx = noise_corr.shape

        # (A) FFT of the noise correlation, hoisted out of the per-source loop
        Rk = np.fft.rfft2(noise_corr)

        for source_id, model in tqdm(self.model_catalog.items()):
            if not hasattr(model, 'flux_err_noisereal_kappa'):
                model.flux_err_noisereal_kappa = {}

            cache = getattr(model, '_kappa_cache', {}).get(band)
            if cache is None:
                model.flux_err_noisereal_kappa[band] = 1.0
                continue

            # (B) Rehydrate H from the sparse cache
            h_vals = cache['h_vals'].astype(np.float64)
            nzy    = cache['nzy'].astype(np.intp)
            nzx    = cache['nzx'].astype(np.intp)
            D      = cache['D']

            H = np.zeros((ny, nx), dtype=np.float64)
            np.add.at(H, (nzy % ny, nzx % nx), h_vals)

            # (C) FFT-based evaluation of h^T R h
            Hk   = np.fft.rfft2(H)
            conv = np.fft.irfft2(Rk * Hk, s=(ny, nx))
            hTRh = float(np.sum(H * conv))

            kappa_sq = hTRh / D
            model.flux_err_noisereal_kappa[band] = np.sqrt(max(kappa_sq, 1.0))

        # (D) free the band's big arrays before touching the next band
        del noise_corr, Rk; gc.collect()
```

Four things to notice, each of them a deliberate memory or performance
choice:

### (A) Hoisting `Rk`

`rfft2(noise_corr)` depends **only on the band**, not on the source. The
old code computed it inside the per-source loop; moving it outside gave
a ~33% speedup (see the project memory entry). It also means we hold
exactly one `Rk` array in RAM during the source loop, not one per
iteration. For each band, `noise_corr` and its FFT together are a few
hundred MB at most; they're the dominant per-band working set.

### (B) Rehydrating H sparsely

The sparse `(h_vals, nzy, nzx)` triple gets scattered into a fresh `H`
via `np.add.at(H, (nzy % ny, nzx % nx), h_vals)`. Two details:

- `np.add.at` (as opposed to `H[nzy, nzx] = h_vals`) is the *unbuffered*
  assignment: it correctly handles any accidental duplicate index, even
  though in practice `nonzero` won't produce duplicates.
- The `% ny` / `% nx` wrap is defensive — `noise_corr` has the shape of
  one image, and for stitched inputs the cache was populated using
  whatever the group's local canvas was. The modulo guards against a
  mismatch rather than forcing the two shapes to agree by construction.

Allocating `H = np.zeros((ny, nx))` each iteration is O(Npix) — it's not
free, but it's the right tradeoff: the alternative is to keep a single
pre-allocated scratch buffer, which complicates cleanup and introduces
state that's easy to get wrong when the inner loop raises. For kappa
computation it's not the bottleneck.

### (C) `h^T R h` via FFT

The three FFT lines are the whole point. `rfft2` → multiply → `irfft2`
is a linear-in-Npix-log-Npix correlation evaluation. After that, `hTRh`
is just `(H * conv).sum()`.

`max(kappa_sq, 1.0)` floors the correction at 1 — we never use kappa to
*shrink* an error bar, only to inflate it. In principle tiny numerical
noise could produce `kappa_sq < 1` for completely isolated sources
where `R ≈ delta`; the floor removes that edge case.

**Subtlety: the FFT runs on the full `noise_corr` canvas every time.**
`H = np.zeros((ny, nx))` is full-image-sized, even though only a few
hundred of its pixels are nonzero for any given source (see §4 on the
dilated-groupmap footprint). The math is correct because `R` is
translation-invariant — `h^T R h` only depends on the *relative* offsets
of the nonzero pixels in `h`, not on where `h` sits on the canvas, so
scattering a sparse h into a full-size canvas and FFT-ing it gives the
right answer. But performance-wise it's wasteful: every source pays
`O(Npix log Npix)` FFT cost regardless of how few pixels it touches.
For a typical group with `N ~ few hundred` nonzero pixels, a direct
`O(N²)` double-sum over nonzero index pairs,

```python
dy = nzy[:, None] - nzy[None, :]
dx = nzx[:, None] - nzx[None, :]
hTRh = float(np.sum(h_vals[:, None] * h_vals[None, :]
                    * noise_corr[dy % ny, dx % nx]))
```

would run in ~10⁴ ops per source instead of ~10⁸ — roughly a 10⁴×
speedup. The FFT path only wins once a single group's nonzero count
exceeds ~`sqrt(Npix log Npix) ≈ 10⁴` pixels, which is rarely hit in
practice. See §8 for when it might be worth switching.

### (D) Free the per-band workspace between bands

After each band's loop, `del noise_corr, Rk; gc.collect()` releases the
large band-level arrays so the next band's `noise_corr` doesn't live
alongside the previous one. With five bands and hundreds of MB each,
skipping this would keep peak RSS ~5× higher than necessary. The
`gc.collect()` is belt-and-braces: `del` drops the reference, but
CPython's generational GC may not immediately return the pages to the
allocator on its own.

Note what is **not** freed: `model._kappa_cache` stays on each source.
That's deliberate — it's tiny per source, and it's what lets a second
`compute_kappa` call be almost free.

## 6. Back-of-envelope memory vs. a naive implementation

For a single IMCOM block with ~5000 detected sources, 5 bands, and
`2108² ≈ 4.4 M` pixels:

| approach                                      | peak working set                 |
|-----------------------------------------------|----------------------------------|
| Naive: store `R_full` once                    | ~150 TB (R_full alone)           |
| Store R as `noise_corr` + per-source `H` dense | ~44 MB × 5000 × 5 ≈ 1 TB         |
| Current: `noise_corr` + sparse cache + per-band del | ~300 MB (one band's `noise_corr` + `Rk`) + ~tens of MB of caches |

The current scheme is literally ~four orders of magnitude leaner than
storing dense templates, and seven orders of magnitude leaner than
materialising the full covariance matrix. Neither alternative is just
"wasteful" — both are impossible on any node you'd actually run on.

## 7. Where the pieces live

- **Build the per-band `noise_corr`:** `FarmerImage.__init__`,
  `image.py:126` — reads the noise realisation cube, estimates a
  smoothed per-pixel sigma, computes `|FFT|²` averaged over realisations,
  and inverse-FFTs to get `r(dy, dx)`.
- **Populate each source's `_kappa_cache`:**
  `SourceGroup._cache_kappa_data`, `_group.py:837` — runs once per
  group, per band, as part of the fit finalisation. This is the
  memory-saving hot spot.
- **Apply the correction:** `FarmerImage.compute_kappa`, `image.py:550`
  — the loop above. Runs once after all groups are fit.
- **Consume the correction:** downstream code in `image.py` /
  `utils.py` scales `flux_err_des` by `kappa` before writing the
  catalog. Search for `flux_err_noisereal_kappa` to follow the trail.

## 8. When it might matter to revisit this

- **If sources span much larger groups than one IMCOM block**, the
  `int16` indices in the cache are no longer enough — raise them to
  `int32` (still 2× smaller than `int64`).
- **If you start modelling stitched canvases bigger than ±32767 px** on
  a side — same issue.
- **If you add per-pixel sigma variation that breaks translation
  invariance** (e.g. gradient-corrected noise models), `noise_corr`
  stops being a sufficient summary and you lose the FFT shortcut.
  You'd have to factor `h = sqrt(invvar) * T` and push the spatially
  varying part into `h` rather than `R`.
- **If you observe `kappa` spending a lot of time in the per-source
  FFT**, the biggest win is to drop the FFT path entirely for typical
  groups and do the `O(N²)` double-sum directly over the ~few-hundred
  nonzero pixels (§5C). That's ~10⁴× faster than FFT on the full
  canvas for compact sources, and only loses to FFT once a single
  group has `N ≳ 10⁴` nonzero pixels — a regime Roman rarely enters.
  A hybrid implementation (direct sum for small `N`, FFT for large
  `N`) is the robust choice.

Otherwise, `compute_kappa` is in a good place: correct, tractable, and
explicit about the tradeoffs that make it tractable.
